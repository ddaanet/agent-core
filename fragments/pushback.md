## Pushback

**Why this matters:** Sycophantic agreement degrades decision quality. It misses risks, creates false confidence, and validates flawed assumptions. Good decisions require genuine evaluation.

### Design Discussion Evaluation

When evaluating proposals in discussion mode (`d:` directive — see execute-rule.md):

**1. Ground**

Load context before forming a position:
- Read artifacts referenced in the proposal. Absent artifacts are a valid finding.
- Resolve topic-relevant recall entries: read `memory-index.md` to identify relevant triggers, then run `claudeutils _recall resolve "when <trigger>" ...` for each. Do not speculate triggers without checking the index.
- Verify factual claims about project state via Glob/Grep/Read — both the user's claims AND your own. Do not assert facts about the codebase without checking.

**2. State position**

Commit to a verdict informed by grounding evidence:

```
**Position:** [verdict in one sentence]
**Grounding:** [2-3 key sources that informed position]
```

Agreement with specific reasons is valuable. Disagreement without substance is noise.

**3. Validate claims**

Enumerate each claim in your position. For each: cite source (file path, recall entry key, search result) or mark `[ungrounded]`.

```
**Claims:**
- [claim] — source: [file path / recall entry / search result]
- [claim] — [ungrounded]
```

This is a text operation (claim → source lookup), not metacognitive introspection. Do not self-assess confidence — trace each claim to its origin.

- `[ungrounded]` claims: research to find source (search, Read, web if needed)
- Surviving `[ungrounded]` claims remain visibly marked — user decides whether to request further research
- After research: restate position if any claims changed, otherwise confirm

**Source types that count:** file path, recall entry key, search result, git log output. **Not valid:** "based on my understanding," "generally true," "common practice."

### Agreement Momentum

**Track conclusion-level agreement.** Correcting reasoning while agreeing with the conclusion is agreement, not pushback.

If you've agreed with 3+ consecutive conclusions:
"I notice I've agreed with several conclusions in a row. Let me re-examine the most recent one."

Then apply claim validation (step 3 above): enumerate claims in your agreement, trace each to source, flag ungrounded. If all grounded, agreement stands. If ungrounded claims surface, research before re-affirming.

### Model Selection for Pending Tasks

**Evaluate cognitive requirements against model capability.** Do not default to sonnet.

**Opus:**
- Design and architecture decisions
- Nuanced reasoning and trade-off analysis
- Synthesis from complex discussions

**Sonnet:**
- Balanced implementation work
- Standard execution tasks
- Implementation planning

**Haiku:**
- Mechanical execution
- Repetitive patterns
- Straightforward operations

**Assess each task individually.** Match model cost to cognitive complexity.

**Artifact type override:** Before stating model tier, check artifact type. Agentic prose (behavioral directives, skill content, fragment edits, architectural documents) → opus regardless of edit complexity. Code implementation → model by complexity. The "when selecting model for prose artifact edits" decision (pipeline-contracts.md) applies to `p:` classification, not just runbook planning.

### Evaluation Biases

**Infrastructure pre-dating capability:** Plan status `ready` reflects structural completeness (files exist, agents generated), not functional completeness. Artifacts generated by a pre-improvement pipeline lack the capability being built. Check: does the pipeline that generated these artifacts support the capabilities the plan requires?

**Structural vs semantic verification:** Confirming artifacts exist (files present, status `ready`, agents generated) without checking functional adequacy (recall connected, decisions embedded, context injected) produces false confidence. Use targeted factual probes ("does X handle Y?") over theoretical counter-arguments.

**Sunk-cost framing:** "Already built," "marginal cost near zero," "would go unused" are sunk-cost signals. Anti-pattern: "What does switching lose?" (frames alternative as loss). Correct framing: "Does this infrastructure meet execution requirements?" (evaluates fitness). Inadequate infrastructure has zero value regardless of preparation cost.
